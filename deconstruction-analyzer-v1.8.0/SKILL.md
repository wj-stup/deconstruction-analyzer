> 专注爆款网文拆书分析，输出结构化拆书报告

## 🚨 避坑速查

> 遇到问题先看这里，秒解常见坑。

| 症状 | 原因 | 一键修复 |
|:-----|:-----|:---------|
| 报告内容为空 / 只有标题没有分析 | 素材搜索失败 + 内置知识未兜底 | 重新运行分析，或手动提供一部同类型作品名称 |
| 报告章节不全（缺第4章/第7章等） | AI 填充时跳过了该章节 | 在对话中要求补充缺失章节，或手动添加 |
| `找不到报告文件` | 报告还没生成就运行了校验 | 先完成一次拆书分析再校验 |
| `文件内容无法正常读取` | 报告文件被其他程序占用或编码不是 UTF-8 | 关闭占用程序，或在编辑器另存为 UTF-8 |
| `没有文件访问权限` | 文件被 Word/编辑器锁定 | 关闭占用该文件的程序后重试 |
| `还没有开始过分析` | 没有先 init 就执行了 add/diff | 先做一次拆书分析初始化 |
| 会话历史丢失 / 重置 | JSON 格式损坏自动重置了 | 正常现象，重新 init 即可 |
| 类型匹配为"通用分析" | 输入太短或不属于内置11大类 | 换更常见的说法（见[Q3](#q3支持哪些网文类型)），通用分析仍可参考 |
| 输入太短（1个字）提示 | 单字无法精确匹配 | 至少2字，如"末世"而非"末" |

## 🚀 快速使用

### ⚠️ 项目目录规则（强制）

**本 Skill 生成的所有文件（拆书报告、会话数据等）必须写入用户指定的项目目录，禁止写入 Skill 自身目录。**

使用前必须遵守以下规则：

1. **首次使用时必须询问用户**：项目文件保存在哪个目录？用户未指定则不可继续——`--project` 已改为必需参数，脚本不会接受空值或默认到 cwd
2. **所有脚本调用传入 `--project {项目目录}`**：session_tracker.py 等写磁盘脚本通过此参数确定输出路径
3. **AI 生成报告落盘到 `{项目目录}/deconstruction_report/report.md`**：不要写到 Skill 安装目录下
4. **项目目录不存在时自动创建**
5. **文件路径构造必须跨平台**：禁止字符串拼接路径，必须使用 `pathlib.Path` 或 `os.path.join()`

脚本参数说明：
- `session_tracker.py`：`--project {项目目录}`（**必需**；所有子命令 init/add/get/list/diff 均需传入）

### 新手最小化启动（2步即可）

1. **告诉 AI 你的项目目录**：`"帮我把文件存到 D:\my_novel"`
2. **说你要拆什么**：`"帮我拆一下都市修仙类"`

AI 会自动运行 type_router 路由类型 → 搜索素材 → 填充报告 → 校验 → 输出到你的目录。

### 基本用法
```
用户说：帮我拆一下"都市修仙"类
```

### 指定作品分析
```
用户说：帮我分析《斗破苍穹》的开篇结构和爽点节奏
```

### 配合 multi-role 完整闭环
```
用户说：拆一下都市修仙类，然后帮我写一本2万字的
```

### 写作问题诊断（新增 🆕，🔧 problem_matcher.py）
```
用户说：我开篇没人看怎么办
  → 调用 problem_matcher.py 匹配症状关键词
  → 输出：根因分析 + 解决方案 + 市场对标数据
```

## 🔬 核心工作流

本 Skill 包含 **2 个工作流**，AI 根据用户意图自动路由：

### Phase -1：写作问题诊断 🩺

当用户说"我卡住了""节奏太慢""开篇没人看"等具体写作问题时，不走拆书分析，直接诊断：

```
Phase -1：写作问题诊断
  │
  ├── Step -1：症状匹配（脚本化 🔧）
  │     解析用户描述 → 提取关键词 → 调用 problem_matcher.py 匹配12类问题
  │     ├── 匹配到具体问题 → 进入 Step -2
  │     └── 未匹配到 → 引导用户描述更具体的症状
  │
  ├── Step -2：根因分析
  │     按 problem_matcher.py 输出的"可能根因"逐项排查
  │     ├── 读取对应素材文件获取诊断清单
  │     └── 让用户逐一回答排查问题
  │
  ├── Step -3：输出方案
  │     技法调整建议（引用 problem_matcher.py 的解决方案）
  │     市场对标数据（引用拆书报告中该类型的分析结论）
  │     优先级建议（快速见效 → 长期改进）
  │
  └── Step -4：可选 → 转入拆书分析
        如果用户还需要深入分析该类型 → 转入 Phase 0
```

### Phase 0：拆书分析（深度拆解）

  ├── Step 1：意图诊断 + 类型路由 🔄（🔧 type_router.py，已整合 intent_mapper）
  │     一次调用同时输出：类型路由 + 意图策略 + 维度权重
  │
  ├── Step 2：情报搜集
  │     搜索该类型爆款 → 确定 3-5 本对标作品
  │
  ├── Step 3：逐本深度拆解 🧬
  │     按类型路由确定的维度权重，进行针对性分析
  │
  ├── Step 4：交叉对比提炼
  │     多书对比 → 提炼共性模板
  │
  └── Step 5：生成报告
        AI 读取 report-template.md 直接填充 → 完整性校验 → deconstruction_report/report.md
```

## 🔬 详细工作流说明

### Phase -1：写作问题诊断

> 当用户描述写作问题（如"节奏太慢""开篇没人看"）时，直接进入诊断流程。

#### Step -1 → 症状匹配（🔧 problem_matcher.py）

调用 `python scripts/problem_matcher.py "{用户描述的问题}"`，输出：匹配结果+置信度、可能根因（2-3个）、解决方案。未匹配则引导用户更具体描述症状后重试。

#### Step -2 → 根因分析

基于 problem_matcher 输出逐项排查：读取素材文件获取诊断清单 → 让用户回答排查问题 → 缩小根因范围。

#### Step -3 → 输出方案

技法调整建议 + 市场对标数据（如有拆书报告则引用） + 优先级排序（快速见效→长期改进）→ 可选转入 Phase 0 深入分析。

---

### Phase 0：拆书分析

#### Step 1 → 意图诊断 + 类型路由（一步完成）

> 原 Step 1（intent_mapper）和 Step 1.5（type_router）已合并。`type_router.py` 内部调用 `intent_mapper.map_intent()`，一次输出同时包含类型路由配置和意图策略。

**用户输入形式**：
- 按类型：`"都市修仙"`、`"悬疑推理短篇"`、`"番茄热榜上的重生类"`
- 按作品：`"分析《斗破苍穹》的开篇结构"`、`"拆解《诡秘之主》的世界观设定"`
- 按平台+榜单：`"起点月票榜前10的玄幻文"`、`"番茄男频阅读榜"`

**AI 行动**：

1️⃣ **一次调用完成意图+路由** — 调用 `python scripts/type_router.py "{用户输入}"`

输出包含：
- **类型路由**：大类/小类/子类匹配 → 维度权重 + 素材引用 + 节奏特征 + 开篇推荐 + 爽点侧重 + 写作模式
- **意图策略**：`intent.intent_id`（A~F）→ 侧重维度 + 报告章节聚焦 + 额外产出
- **权重合并**：当意图为非"全面分析"时，`intent.analysis_weight_shift` 逐 key 覆盖大类默认权重

```
用户说："末世种马种田"
  ↓
python scripts/type_router.py "末世种马种田"
  ↓
输出：
  类型路由：末世/废土大类 → 权重{节奏:5,金手指:5,...}
  意图策略：F(全面分析) → 标准9章节
  素材引用：[素材-爽点库.md, 素材-问题诊断.md, ...]
```

2️⃣ **意图澄清**（仅当用户未明确意图时，AI 可反问"你想从拆书分析中得到什么？"并给出 A~F 六个选项，对应关系见下方映射表）

3️⃣ **交叉类型识别** — 检查 `type_router` 输出的 `cross_type_info` 是否非空：
```
用户说："末世言情"
  ↓ type_router 返回 cross_type_info: {primary: 末世, secondary: 言情}
  ↓ 同时分析两个大类的特征
  ↓ 在报告中增加"交叉地带专项分析"章节
```

**意图 → 分析策略映射**：

| 用户意图 | intent_id | Step 3核心侧重 | Step 5报告侧重 | 额外产出 |
|:---------|:---------:|:--------------|:---------------|:---------|
| 📊 市场行情 | A | 类型总览+竞品扫描 | 章节一+章节七 | 热度对比表 |
| 🏗️ 结构模板 | B | 全篇结构+章节分布 | 章节二+章节三 | 可复用的结构框架 |
| 🎣 开篇写法 | C | 前3章逐句分析 | 章节五 | 开篇对照表 |
| 💎 金手指 | D | 金手指类型+升级路径 | 章节六 | 金手指卡牌模板 |
| ⚠️ 避坑 | E | 读者差评+逻辑漏洞 | 章节七 | 避坑清单 |
| 🔬 全面分析 | F | 所有6维度 | 标准9章节 | 完整报告 |

**输出**：路由配置 + 意图策略 → 直接传递给 Step 2

---

#### Step 2 → 情报搜集（联网）

使用 web 搜索能力，按 [搜索策略SOP.md](references/搜索策略SOP.md) 执行：

1. 第一轮搜索：类型+平台定位 → 确定热门作品候选池
2. 第二轮搜索：对标作品深挖 → 获取每本的简介/热度/读者评论
3. 第三轮搜索：写作技法补充 → 开篇结构/常见雷区等

**核心规则**：
- 确定 3-5 本最具代表性的对标作品
- 每本按「最低信息标准」判定达标（详见 SOP 第3节）
- 无法获取正文时，使用 `passage_extractor.py --mode infer` 处理搜索片段
- 搜索失败时使用内置知识库降级

**输出**：对标作品清单（每本含简介+热度数据）

---

#### Step 3 → 逐本深度拆解（类型自适应）

根据 Step 1 类型路由确定的维度权重，对每本对标作品进行针对性分析。

**通用6维度（所有类型的基础分析框架）**：

##### 维度 A：结构分析
- 黄金三章结构（每章内容/字数/钩子）
- 全篇章节分布（Act 1/2/3 划分）
- 章节字数范围

##### 维度 B：爽点节奏
- 爽点类型统计（根据类型不同，爽点类型不同）
- 爽点密度（每几章一个）
- 情绪曲线（高潮低谷分布）

##### 维度 C：人物体系
- 主角类型（根据类型不同，原型库不同）
- 主要配角类型
- 人物关系网特点

##### 维度 D：金手指/核心设定
- 金手指类型（根据类型不同，侧重点不同）
- 能力体系（升级途径/限制条件）
- 世界观规则

##### 维度 E：开篇钩子
- 开篇方式（根据类型不同，有效开篇公式不同）
- 前300字钩子设计
- 第一章结尾悬念

##### 维度 F：风险识别
- 读者负面评论中反映的问题
- 类型常见逻辑漏洞
- 写作雷区

**额外维度（根据类型路由选择性开启）**：

> 额外维度完整映射表见 [genre-taxonomy.md](references/genre-taxonomy.md)「额外维度映射」章节，权威数据源为 `taxonomy_data.py TAXONOMY` 各子类节点的 `extra_dimensions` 字段。

**分析维度权重分配规则**：
```
总分析资源 = 100%
  通用6维度：    占 60%（各维度按类型权重表分配）
  额外维度：     占 20%（按类型路由开启的维度数均分）
  范文段落精析： 占 10%（原文引用 + 技法标注）
  创新点发现：   占 10%（该作品区别于同类的独特之处）
```

**范文段落精析**：使用 `scripts/passage_extractor.py` 提取对标作品关键段落（开篇/高潮/金手指首次展示），做原文引用+技法标注。无法获取原文时用 `--mode infer` 从搜索片段推断，报告中必须附加 ⚠️推断·置信度 标注。

**脚本工具**：`scripts/passage_extractor.py` 可对提取的段落做自动化分析

**输出**：每本作品的类型自适应拆解笔记

---

#### Step 4 → 交叉对比提炼

将 Step 3 的多本分析结果进行对比：

1. **共性提取**：哪些结构/节奏/人物模式在多部作品中出现 → 形成类型模板
2. **差异分析**：各作品的差异化创新点是什么 → 提供创作灵感
3. **趋势判断**：近期新作在哪些方面有变化 → 把握市场方向
4. **最佳实践**：哪种处理方式最受读者认可 → 形成推荐方案

**输出**：交叉对比结论 + 类型模式模板

---

#### Step 5 → 生成报告

分两步执行：

**第 1 步：AI 读取模板直接填充**

1. 读取 `references/report-template.md` 获取9章节结构和必含字段要求
2. 基于 Step 3/4 的拆解分析笔记，AI 逐一填充 一~九 节的分析内容
3. 确保填充内容与 Step 1 路由输出的维度权重、意图策略一致
4. 按模板要求写入 YAML 元信息头（含 `适用写作模式`，由 type_router 的 `writing_mode` 字段确定）

**第 2 步：完整性校验**

1. 调用 `python scripts/report_validator.py "{项目目录}/deconstruction_report/report.md"`
2. 检查输出：
   - `valid: true` → 校验通过，报告完成
   - `valid: false` → 根据校验报告中的错误/警告逐项修复
3. 修复后再次校验，直到通过（⚠️ 最多重试 **3 次**；3 次仍未通过时，将校验报告的未通过项整理为清单供用户手动处理，不要无限循环）

**输出**：最终拆书报告 `deconstruction_report/report.md`

> ⚠️ **必须告知用户文件位置**：报告生成后，必须向用户明确说明文件的保存路径和文件名，例如：
> 📄 拆书报告已生成，保存在：`{项目目录}/deconstruction_report/report.md`
> 如用户未指定项目目录，则说明默认路径。

如果用户需要，还可以：

用户说：帮我专门拆一下这类小说里的"感情线"安排
→ 在标准 6 维度之外，增加感情线专项分析
```

#### 单本精拆
```
用户说：把《斗破苍穹》从头到尾完整拆一遍
→ 逐章拆解 + 整体结构分析 + 角色成长弧线
```

#### 类型对比
```
用户说：对比分析"都市修仙"和"玄幻修仙"的差别
→ 两个类型分别拆解 → 交叉对比报告
```

#### 迭代拆书（增量分析）

> session_tracker.py 仅在**增量分析**（用户追加分析需求）时启用。单次分析不调用，关键信息直接记录到报告 YAML 头。

```
用户第一次说："帮我拆一下都市修仙类"
→ 单次分析，不调用 session_tracker → 输出完整报告

用户第二次说："这个类型里的重生流变种再深挖一下"
→ AI 检测到已有报告 → 启用 session_tracker 增量模式
→ 调用 scripts/session_tracker.py add --project {项目目录} --works "新作品X" --focus "重生流变种"
→ 在已有报告基础上做增量分析
→ 输出差异报告：哪些结论不变、哪些需要更新、新增了什么

用户第三次说："把口碑最好的3本单拎出来精拆"
→ session_tracker 继续追加
→ 调用 scripts/session_tracker.py add --project {项目目录} --works "作品X" "作品Y" "作品Z" --focus "精拆"
→ 增量输出：3本作品的逐章深度拆解

底层机制：
  - 首次分析跳过 session_tracker，关键信息（类型/对标作品/分析时间）记录到报告 YAML 头
  - 增量分析时调用 python scripts/session_tracker.py add --project {项目目录} 记录上下文
  - 增量分析先调用 python scripts/session_tracker.py get --project {项目目录} 检查历史
  - 如有历史，先运行 python scripts/session_tracker.py diff --project {项目目录} --old <上一轮> --new <本轮>
    了解"增量"是什么，只分析新内容
  - 项目目录下的 deconstruction_report/.session_history.json 记录全链路
```

| 序号 | 章节标题 | 标记 |

| 八 | 多角色速查表 | 📊 |

### 文风DNA标签规范（第九章核心产出）

从多本对标交叉对比中提炼3-5个类型级文风DNA标签，每个标签采用四段式结构：

```
标签N：[特征名称]（类型级）
  定义：（一句话精准描述该类型的典型文风特征）
  量化区间：（从多本标品中统计的数值范围，如"句长12-18字，动作占比38-45%"）
  创作提示：（写该类型时默认启用，偏离需有意识设计）
  偏离案例：（某作品偏离此特征后的市场反馈）
```

### 量化指标卡规范

每个分析章节（二/三/四/五/六/七）末尾必须附量化指标卡：

```
📊 **量化指标卡**
| 指标 | 数值 | 健康阈值 | 状态 |
|------|------|----------|:----:|
| 爽点密度 | 每4.2章一次 | 前20章每3章≥1次 | ✅ |
```

各章节对应指标见 [report-template.md](references/report-template.md) 第4节。

### 可视化图谱规范

报告必须包含 3 类一级图谱（Mermaid格式），推荐包含 2 类二级图谱。详见 [report-template.md](references/report-template.md) 第5节。

### 推断透明度规范

基于 web 搜索片段而非完整正文的分析，必须标注：
- `⚠️推断·置信度：高/中/低`
- `⚠️推断·原因：...`
- 素材完整度≤60%时，宏观结论必须附加推断标注
- YAML 头必须填写 `素材完整度` 字段

### 输出完整性自检

报告生成后由 `report_validator.py` 自动校验。AI 额外检查：量化指标卡至少覆盖2个章节 + 至少3类Mermaid图谱 + 第九章含文风DNA标签。

## ❓ 常见问题

### Q1：首次使用怎么开始？

只需两步（详见上方「新手最小化启动」）：
1. 告诉 AI 你的项目目录（如 `D:\my_novel`）
2. 说你要拆的类型（如 `帮我拆一下末世废土类`）

AI 会自动完成类型路由→素材搜索→报告填充→完整性校验，报告保存在 `{项目目录}/deconstruction_report/report.md`。

### Q2：项目目录必须指定吗？

**必须**。所有输出文件（报告、会话数据）都写入项目目录，脚本不接受空路径或默认到当前工作目录。这是为了避免文件散落到不可控的位置。

如果目录不存在，脚本会自动创建。

### Q3：支持哪些网文类型？

内置 **11 大类 → 33 小类 → 84 子类** 三级分类体系，覆盖主流网文类型：

| 大类 | 小类举例 |
|------|---------|
| 1.玄幻 | 传统玄幻、系统玄幻 |
| 2.修仙/修真 | 传统修仙、重生修仙、穿越修仙、科技修仙 |
| 3.都市 | 都市异能、都市生活 |
| 4.末世/废土 | 末世重生、末世天灾、末世变异、废土生存 |
| 5.科幻 | 星际科幻、时间科幻、AI/赛博 |
| 6.悬疑/推理 | 推理侦探、恐怖悬疑、无限流/副本 |
| 7.游戏 | 游戏穿越、电竞、网游/MMO |
| 8.历史/架空历史 | 历史穿越、架空历史 |
| 9.言情 | 现代言情、古代言情、玄幻言情 |
| 10.轻小说/二次元 | 日常轻小说、同人/衍生、脑洞/设定系 |
| 11.无限流/生存游戏 | 经典无限流、对抗无限流、解谜无限流、建设无限流 |

输入不匹配也可以——同义词扩展和交叉类型推断会自动处理（如"末日伪人囤货流"→末世+恐怖双类型）。

### Q4：报告校验失败怎么办？

AI 在生成报告后通常会自动调用 `report_validator.py` 校验，最多重试 **3 次**补全缺失内容。3 次校验后仍未通过时：

1. 查看校验报告中的 ❌ 项——它会告诉你缺了哪个章节/字段
2. 常见失败原因：
   - **缺章节**：AI 填充时跳过了某个章节 → 手动补全该章节内容
   - **占位符未填充**：报告中残留 `[LLM:FILL]` → AI 重新填充该段落
   - **子项不足**：某章节下 ### 子标题不够 → 补充分析点
3. 修复后手动运行 `python scripts/report_validator.py "{项目目录}/deconstruction_report/report.md"` 确认

### Q5：怎么和 multi-role（AI创作天团）配合？

全自动：先用本技能生成报告 → 切到 multi-role 在同一项目目录启动创作 → `quick_start.py` 自动检测并注入正典。也可手动指定 `--deconstruction-report {报告路径}`。拆书报告是**可选依赖**——没有也能正常使用。

### Q6：我的类型很冷门/找不到怎么办？

1. **先试同义词**：比如"异兽流"可能归属"都市异能"，"灵异文"可能归属"恐怖"→"规则类恐怖"
2. **尝试拆分描述**：输入复合类型如"末世+系统流"，type_router 会自动识别交叉类型
3. **如果确实不在分类中**：type_router 返回 `matched: false` 时会附带建议（如"最接近的已有类型"），可按建议选一个相近类型分析，冷门类型的分析精度会偏低

> 如果某个类型反复匹配不到，欢迎反馈，分类体系会持续扩充。

### Q7：搜索不到素材/搜索失败会影响报告吗？

不会导致报告无法生成。素材搜索失败时脚本自动降级到内置知识库，报告仍能正常产出，但以下方面可能受影响：

- 章节分析深度降低（缺少具体作品引用和原文片段）
- 推断标注增多，置信度降低
- 开篇钩子和金手指分析趋于通用化（缺少类型特异性的范例）

**建议**：搜索失败后，可手动提供一部同类型作品的名称或核心设定，AI 会基于内置知识补充分析。

### Q8：输入特殊字符或太短怎么办？

脚本会自动清洗输入中的零宽字符和控制字符，无需手动处理。如果输入太短（只有1个字），会提示"请多说几个字"。建议输入至少2个字以上的类型关键词。

### Q9：运行出错了怎么办？

v1.8.0 起所有报错信息已翻译为通俗易懂的中文提示。遇到报错时：

1. **先看避坑速查表**（上方 🚨 避坑速查），90% 的问题有对应解法
2. **查看报错提示**——现在不再显示技术术语（如 FileNotFoundError），而是直接告诉你"文件未找到，请检查路径"
3. **文件被占用时**——脚本会自动重试 2 次（间隔1秒），如果仍失败才报错

| 报错提示 | 怎么办 |
|----------|--------|
| `请告诉我你想分析什么类型` | 没有输入类型，说一句"帮我拆一下XX类"即可 |
| `你输入的XX太短了` | 多写几个字，比如'末世重生'而不是仅'末' |
| `文件未找到，请检查路径` | 确认文件路径拼写正确，报告是否已生成 |
| `没有文件访问权限` | 关闭占用文件的程序（如 Word），1 秒后自动重试 |
| `文件编码无法识别` | 在编辑器中另存为 UTF-8 格式 |
| `文件内容格式有误，无法解析` | 文件可能损坏，让 AI 重新生成 |

如果以上方法都无法解决，可以把完整的报错信息发给AI，AI会帮你排查。

## ⚠️ 重要原则

1. **诚实标注来源**：拆书分析中引用具体作品时，清晰标注来源
2. **不抄袭**：分析目的是提炼模式模板，而非鼓励抄袭
3. **尊重版权**：只分析公开可见的结构性信息，不搬运原文大段内容
4. **时效性**：网文类型热度变化快，标注分析时间
5. **报告可编辑**：输出报告后用户可手动修改补充
6. **graceful fallback**：网络搜索失败时使用内置知识库降级
